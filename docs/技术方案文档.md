# UniLife OS — 大学生 AI 校园生活助手

## 技术方案文档

> **赛道四：生活帮手 Agent**
> 宿舍、食堂、图书馆、实习通勤……大学生活的每一刻，都值得被智能守护。

---

## 一、项目概述

UniLife OS 是一个专为大学生打造的 **AI 校园生活操作系统**。用户通过自然语言对话，即可完成课程管理、财务记账、健康打卡、待办追踪、旅行规划五大模块的全部操作——无需学习任何界面，说一句话就能搞定。

系统内置 **24 个 Agent 工具**，基于 DeepSeek-V3 大模型的 function calling 能力实现自主决策：AI 理解用户意图后，自动选择工具、执行操作、将结果转化为自然语言回复，最多支持 5 轮连续推理。同时提供实时数据看板和侧边栏快速操作面板，兼顾"对话式交互"与"传统 GUI"两种使用习惯。

支持 PWA 添加到手机主屏幕，获得接近原生 App 的使用体验。

## 二、痛点与需求分析

我们在校园调研中发现，大学生日常生活面临**信息碎片化**和**工具割裂**两大痛点：

| 场景     | 痛点                                                 | UniLife 的解法                                  |
|----------|------------------------------------------------------|------------------------------------------------|
| 课程管理 | 课表散落在教务系统、截图、Excel 中，临时调课难以同步   | 自然语言增删改课表，实时侧边栏展示今日课程      |
| 财务记账 | 想记账但打开 App 太麻烦，坚持不了三天                 | 对话记账（"奶茶 18 块"），自动分类 + 预算预警   |
| 健康管理 | 步数、睡眠、喝水分散在不同 App，无法形成整体画像      | 一站式健康面板 + 7 天趋势图 + 智能提醒          |
| 待办追踪 | 作业、社团、生活琐事混在一起，容易遗漏紧急事项        | 优先级分级 + 过期自动清理 + AI 主动提醒         |
| 出行规划 | 约同学出去玩，行程预算全靠脑子记                      | 完整旅行计划 CRUD + 行程时间线 + 必带清单       |

**核心洞察**：大学生不需要又一个"功能堆叠"的 App，而需要一个**懂上下文、能主动思考的智能助手**——知道你今天有什么课、这个月花了多少钱、还有几天考试，然后在你需要的时候，用最自然的方式帮你搞定。

## 五、核心技术方案

### 5.1 Agent 自主决策循环

这是系统的核心引擎。当用户输入一句话后，AI 不只是生成回复——它会**自主判断是否需要调用工具、调用哪些工具、如何组合结果**，最多进行 5 轮连续推理：

```
用户输入: "帮我记一笔奶茶 18 块，顺便看看这个月花了多少"
  |
  v
DeepSeek-V3 意图识别
  |
  | 决策: 需要调用 2 个工具
  v
Round 1: record_expense
  ├── item = "奶茶"
  ├── amount = 18
  └── category = "餐饮"
  |
  | 工具结果反馈给模型
  v
Round 2: query_finance
  |
  | 工具结果反馈给模型
  v
生成最终回复:
  "记好了~ 奶茶 ¥18 已记入餐饮类。
   这个月已经花了 ¥1668，预算用了 83%，
   剩下 3 天建议每天控制在 ¥55 以内哦~"
```

**关键实现细节**：

```python
def chat_agent(messages, tools, execute_tool_fn):
    working_messages = list(trim_messages(messages))
    tool_call_log = []

    for _round in range(MAX_TOOL_ROUNDS):      # 最多 5 轮
        response = _call_with_tools(working_messages, tools)
        assistant_msg = response.choices[0].message

        if not assistant_msg.tool_calls:         # 无工具调用 → 返回
            return assistant_msg.content, tool_call_log

        # 有工具调用 → 执行并继续循环
        for tc in assistant_msg.tool_calls:
            result = execute_tool_fn(tc.function.name, json.loads(tc.function.arguments))
            working_messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})
            tool_call_log.append({"name": tc.function.name, "result": result})

    # 超过 5 轮，做一次无工具调用获取总结
    return get_client().chat.completions.create(messages=working_messages).choices[0].message.content, tool_call_log
```

### 5.2 增量覆盖持久化设计

这是数据层的核心设计模式，解决了**原型演示与数据持久化的矛盾**：

```
mock_data (基础数据)                persistence (用户增量修改)
  ├── 10 门课程                       ├── deleted_ids: [4]       ← 删了体育课
  ├── 21 条消费记录                   ├── extra_txns: [..]       ← 新记了一笔
  ├── 7 条待办                        ├── health: {water: +3}    ← 今天喝了3杯
  └── 8 个行程站点                    └── course_updates:        ← 高数换教室
                                            {1: {location:..}}
          \                          /
           +--------合并--------+
                    |
                    v
           get_*() 合并函数
             ├── 过滤已删除
             ├── 应用修改覆盖
             ├── 追加新增项
             └── 重算统计值
```

以课表为例，合并逻辑为三步：

```python
def get_schedule():
    base = [...]                                 # 10 门 mock 课程
    deleted_ids = get_deleted_course_ids()
    schedule = [c for c in base if c["id"] not in deleted_ids]  # Step 1: 过滤删除
    updates = get_course_updates()
    for c in schedule:                                           # Step 2: 应用修改
        if str(c["id"]) in updates:
            c.update(updates[str(c["id"])])
    schedule.extend(get_extra_courses())                         # Step 3: 追加新增
    return schedule
```

**设计优势**：
- 基础数据始终可用（即使 JSON 文件损坏，重置后即恢复）
- 用户修改量极小（只存差异），JSON 文件体积可控
- 每种数据类型（课表/财务/健康/待办/旅行）统一采用此模式，架构一致性强

### 5.3 动态上下文注入

每次对话时，系统将用户的**实时数据汇总为结构化摘要**，注入 System Prompt：

```python
def build_system_prompt(user_context):
    return f"""
    # 你是 UniLife，一个专为大学生打造的 AI 生活助手
    ## 你可以看到用户的以下实时数据：
    ### 📅 课程信息
    {user_context['schedule_summary']}      ← "今天(周三)有2节课：08:30 高数..."
    ### 💰 财务状况
    {user_context['finance_summary']}       ← "本月预算2000元，已花费1668元(83%)..."
    ### 🏥 健康状态
    {user_context['health_summary']}        ← "今日步数4523/8000，已有5天未运动..."
    ### 📝 待办事项
    {user_context['todo_summary']}          ← "待办4项，其中1项紧急！"
    ### 🔔 当前提醒
    {user_context['alert_summary']}         ← "预算告急、考试倒计时、运动提醒..."
    ## 工具使用指引
    [24 个工具的调用时机说明]
    """
```

这意味着 AI **每一轮对话都感知着用户的最新状态**——不是等用户问"我花了多少钱"才去查，而是在用户说"今天好累"的时候，它已经知道你昨晚只睡了 6.5 小时、已经 5 天没运动了，可以给出有温度的关怀建议。

## 六、鲁棒性与安全设计

### 6.1 数据安全

| 措施           | 实现                                                        | 防护目标                       |
|----------------|-------------------------------------------------------------|--------------------------------|
| 原子写入       | `tempfile.mkstemp()` → 写入 → `os.replace()` 原子重命名     | 防止写入中断导致 JSON 损坏     |
| 临时文件清理   | 写入失败时 `os.unlink(tmp_path)` 清理残留                    | 防止磁盘垃圾堆积               |
| 静默降级       | `OSError` 捕获后 `stderr` 输出，不崩溃                       | 磁盘满/权限不足时保持可用       |
| XSS 防护       | 用户输入经过 `html.escape()` 后再传入 `unsafe_allow_html`    | 防止恶意脚本注入               |
| HTML 标签剥离  | 提醒信息注入 LLM 前用 `re.sub(r"<[^>]+>", "")` 清理         | 防止 HTML 污染模型上下文        |

### 6.2 输入校验

所有 Agent 工具均内置边界校验，防止 LLM 生成异常参数：

```python
# 金额：0 < amount ≤ 100,000
# 步数：0 ≤ steps ≤ 200,000
# 睡眠：0 ≤ hours ≤ 24
# 运动目标：3 ≤ goal ≤ 7（枚举约束）
# 消费项目：非空 + strip() 清洗
# 课程删改：先按 ID/名称查找验证存在，再执行操作
```

### 6.3 时间自动化

| 机制       | 触发条件                  | 行为                         |
|------------|---------------------------|------------------------------|
| 跨天重置   | `override_date ≠ 今天`    | 喝水、步数、睡眠、运动打卡归零 |
| 跨周重置   | `week_start ≠ 本周一`     | 运动周计数归零                |
| 过期清理   | `deadline < 今天 - 7天`   | 待办自动从列表移除            |
| 考试过滤   | `days_left < 0`           | 已过期考试不再显示            |
| 防重复打卡 | `exercise_today == True`  | 同日运动打卡不重复计数        |

### 6.4 Agent 安全边界

- **最大推理轮次**：`MAX_TOOL_ROUNDS = 5`，防止无限工具调用循环
- **上下文裁剪**：`trim_messages()` 保留 system + 最近 20 条消息，防止超出模型窗口
- **对话持久化上限**：最多保存 50 条历史消息，防止 JSON 文件膨胀
- **LLM 类型兼容**：`int(args["task_id"])` 强制转换，兼容模型返回字符串的情况
- **模糊匹配防歧义**：课程/行程按名称查找时，多个匹配返回提示而非随机选择

## 七、人机交互设计

### 7.1 视觉设计语言

- **主色调**：紫蓝渐变（`#667eea → #764ba2`），统一贯穿 Header、Tab、侧边栏
- **毛玻璃效果**：指标卡片采用 `backdrop-filter: blur(6px)` + 半透明背景，现代感强
- **Light/Dark 自适应**：CSS 使用 `background-image` 半透明叠加（而非 `background` 覆盖），保留 Streamlit 原生底色，**单套 CSS 同时适配两种主题**
- **胶囊式 Tab 导航**：选中态为紫蓝渐变填充 + 阴影，非选中态为透明边框

### 7.2 交互反馈机制

- **Toast 延迟显示**：解决 Streamlit `st.rerun()` 会清除 toast 的问题——消息暂存 `session_state`，下次渲染时在 `main()` 顶部弹出
- **工具调用可视化**：Agent 调用工具时实时显示 `st.status("🤔 思考中...")`，每个工具结果可展开查看
- **动态欢迎语**：首次进入时根据当前课程、财务、待办、健康数据生成个性化问候

### 7.3 移动端适配

- PWA standalone 模式隐藏浏览器 UI，获得原生 App 体验
- 图表禁用缩放手势（`dragmode=False`、`scrollZoom=False`），防止手机端误触
- 聊天容器自适应高度（`calc(100vh - 280px)`），输入框始终可见

## 八、效率优化

| 优化点         | 实现方式                                    | 效果                                         |
|----------------|---------------------------------------------|----------------------------------------------|
| API 客户端复用  | 模块级单例 `_client`，延迟初始化              | 避免每次对话创建新连接，复用 HTTP 连接池       |
| 上下文裁剪     | 只保留 system + 最近 20 条消息                | 减少 token 消耗，加快响应速度                  |
| 增量持久化     | 只存用户修改的差异，非全量写入                 | JSON 文件体积极小（通常 < 10KB）               |
| 输出截断检测   | `finish_reason == "length"` 时提示用户        | 避免无提示的内容丢失                           |
| 模型参数调优   | `temperature=0.7`，`max_tokens=1024`         | 平衡创意性与稳定性，控制响应长度                |
